算法刷题方法论记录（二）：滑窗、矩阵与惯性转换（结构升级版：6 类状态变量贯穿 9 题）

我刷题最大的痛点不是“不会做”，而是：

  • 我能靠思维惯性写出一个版本（能过部分样例）
  • 但写不出稳定、可复用的最优模板
  • 常卡在实现细节：循环怎么写、边界怎么吸收、O(1) 空间怎么做到

这篇文章我做了一次“结构升级”：不再按题目零散记录，而是按 6 类最常见的“状态变量 / 不变量” 来组织。因为我发现：
同一类状态变量，会在不同题里反复出现；你真正要背的是“状态更新方式”，不是题解。

下面用 9 道题串起来（每题都有：背景→我原来想干什么→写成了什么→哪里错/慢→怎么转换→最简代码）。

总纲：6 类状态变量（你真正要背的模板）

  1. last_index：去重窗口（“上次出现位置”）
  2. diff + nonzero：窗口等价/匹配（“差分向量 + 非零维度数”）
  3. need + missing：覆盖窗口（“需求表 + 总缺口”）
  4. running_end：区间合并（“当前合并段右端”）
  5. prefix / suffix：滚动乘积/和（“不变量吸收边界”）
  6. reuse input as marker：空间压缩（“复用输入当标记区”）

你后续刷题的目标：
先写惯性代码 → 把它映射到这 6 类之一 → 按该类的“状态更新模板”重构为最优。

1) last_index：LeetCode 3 — 无重复字符的最长子串

题意

给定字符串 s，返回不含重复字符的最长子串长度。

背景：我最开始怎么想（惯性）

滑动窗口 [l, r]：

  • r 不断右移
  • 如果 s[r] 在窗口里出现过，就移动 l

所以我需要一个结构记录：字符上次出现的位置。

我当时想用 dict.get() 干什么？

目的：判断字符是否出现过（存在性判断）。
我写成：

会把 “出现过且位置为 0” 误判成 “没出现过”。 误判后果：l 不移动，窗口错误包含重复字符，答案可能变大/变错。

惯性 → 最优转换（机械步骤）

  • 转换 1：存在性判断不看 value 真值，改为看 key 是否存在

    • if last.get(ch): → if ch in last:

  • 转换 2：左指针不允许回退

    • l = last[ch] + 1 → l = max(l, last[ch] + 1)

2) diff + nonzero：LeetCode 438 — 找到字符串中所有字母异位词

题意

给定 s 和 p，返回 s 中所有与 p 互为异位词的子串起始索引。

背景：我最开始怎么想（惯性）

固定长度窗口 m=len(p)，每次取 s[i:i+m]：

  • 惯性方案：排序后比较是否相等

排序能做，但慢，因为每个窗口都要做一次“重活”。

最优思路一句话（同思路优化，不换题型）

异位词判定不需要排序，只需要比较 26 个字母频次是否一致；窗口滑动时只做 in/out 两次增量更新。

我当时想用 diff_sum/nonzero 干什么？

我想避免每次都扫描 diff[26] 判断是否全 0，于是维护状态量：

  • diff[c] = count_window[c] - count_p[c]
  • nonzero = diff 中非 0 的维度数
  • 若 nonzero == 0，窗口即为异位词

我为什么总写错？写错了什么？（逻辑链补全）

我写错的目标：维护 nonzero 这个状态量，使它始终等价于“非 0 维度数”。 我写成了什么：只在“变更后变成 0”时 nonzero -= 1，漏掉“从 0 变成非 0”时应该 nonzero += 1。

错误形式（典型）：

为什么错？

  • 如果更新前 diff[idx]==0，更新后变成 1/-1，nonzero 应该 +1
  • 你没加，nonzero 失真
  • 后果：nonzero==0 会误报/漏报匹配

最关键的修复模板：跨 0 更新（必须背）

惯性 → 最优转换（机械步骤）

  • 排序比较 → 频次差分向量
  • 每次重算频次 → 只做 in/out 两次更新
  • 每次扫 26 判断全 0 → 维护 nonzero（跨 0 更新）

最简代码模板

3) need + missing：LeetCode 76 — 最小覆盖子串

题意

返回 s 中最短子串，使其包含 t 的每个字符（含重复）。无解返回空串。

背景：我最开始怎么想（惯性）

滑窗 [l, r]：

  • r 扩张直到“覆盖 t”
  • l 收缩到极致得到候选最短
  • 取全局最短

核心难点：“覆盖”怎么判断。

我当时想用 need 干什么？

我定义：

  • need[c] 表示窗口为了覆盖 t 还缺多少个 c
  • 直觉上：如果所有 need[c] <= 0 就覆盖

但“每次遍历 need 判断是否覆盖”会引入额外循环/难写的逻辑分支。

最优思路一句话（同思路优化）

把“是否覆盖”从“遍历判断”重构成一个变量 missing：missing==0 就覆盖。

我当时产生的疑问（为什么会问）

收缩时我会移走 y = s[l]，写：

我担心： 如果 y 不在 t 中，会不会也让 need[y] > 0，从而误判“开始缺字符”，导致收缩过早停止？

这里“误判”的含义是： 把“不在 t 中的字符”误判成“t 里缺的字符”，导致窗口被错误地判为不覆盖。

为什么不会误判？（把逻辑说清）

定义：

  • need[c] = count_t[c] - count_window[c]

若 c 不在 t 中，则 count_t[c] = 0：

  • 当 c 进入窗口，count_window[c] 增加 ⇒ need[c] 变成负数（<=0）
  • 当 c 离开窗口，need[c] += 1 最多回到 0，不可能 >0

所以：

对不在 t 的字符，need 永远不会变成正数； need[y] > 0 只可能表示“窗口真的开始缺 t 需要的字符”。

惯性 → 最优转换（机械步骤）

  • 覆盖判定“遍历 need” → 单变量 missing
  • 每次扩大/收缩 → 对应更新 need 与 missing
  • 形成模板：右扩到 missing==0，左收缩到刚好 missing>0

最简代码模板

4) running_end：LeetCode 56 — 合并区间

题意

合并所有重叠区间，返回合并后的区间列表。

背景：我最开始怎么想（惯性）

排序后比较相邻区间：

  • 如果 intervals[r] 与 intervals[r-1] 不重叠，就断开并输出
  • 否则合并

我错在哪里？（本来想干什么→写成了什么→错在哪）

我本来想判断“当前合并段是否断开”，但我写成了“只看相邻原始区间是否重叠”，并用 intervals[r-1][1] 当作右端。

问题是：右端不应该是“前一个原始区间的 end”，而应该是“当前合并段的最大 end”。

反例：

  • [1,10] [2,3] [4,5] 相邻看 [2,3] 与 [4,5] 不重叠会断开，但它们其实都在 [1,10] 内。

最优思路一句话（同题型，但需要换“状态”）

维护 active interval 的 running_end：后续区间只会延长右端，不会影响左端。

惯性 → 最优转换（机械步骤）

  • “看相邻” → “看 next.start 是否 > running_end”
  • “右端用 intervals[r-1]” → “右端用 max(running_end, next.end)”

最简代码模板

5) prefix / suffix：LeetCode 238 — 除自身以外数组的乘积

题意

返回数组 ans，其中 ans[i] 等于 nums 中除 nums[i] 外所有元素的乘积，不使用除法。

背景：我最开始怎么想（惯性）

构造前缀数组/后缀数组，再拼：

  • pre[i] 表示到 i 的乘积
  • suf[i] 表示从 i 的乘积 然后 ans[i] = pre[i-1] * suf[i+1]

为什么惯性写法会让代码难写？

因为你会被下标绑死，出现大量边界分支：

  • i=0 没有 i-1
  • i=n-1 没有 i+1 于是 if i==0 / i==n-1 一堆。

最优思路一句话（同思路优化：数组 → 状态）

把“前缀/后缀数组”重构为两个滚动状态变量 pre/suf，用乘法单位元 1 吸收边界。

惯性 → 最优转换（机械步骤）

  • 前缀数组 → pre 滚动变量（不变量：pre 等于当前位置左侧乘积）
  • 后缀数组 → suf 滚动变量（不变量：suf 等于当前位置右侧乘积）
  • 边界 if → 用单位元 1 消掉

最简代码模板

6) 原地哈希桶：LeetCode 41 — 缺失的第一个正数

题意

给未排序数组 nums，找最小缺失正整数。要求 O(n) 时间且 O(1) 额外空间。

背景：我为什么会“完全没思路”

这题把两条惯性路线都封死了：

  • set：空间 O(n) 不允许
  • 排序：时间 O(n log n) 不允许

所以必须换模型：数组本身就是哈希桶。

最优思路一句话（与惯性完全不同：必须换思路）

把值 x（1≤x≤n）放到下标 x-1 上（原地交换）；最后第一个 nums[i] != i+1 的 i，就是答案 i+1。

关键实现细节：为什么交换后不能 i++？

你交换一次后，nums[i] 换来了一个新值，这个新值可能也需要继续被放到正确位置。 所以只有当当前位置“稳定”（无效值 / 已在正确位置 / 会重复）时才 i++。

7) reuse input as marker：LeetCode 73 — 矩阵置零（O(1) 空间）

题意

给矩阵 matrix，如果某个元素为 0，则将其所在行和列的所有元素设为 0。要求原地 O(1) 额外空间。

背景：我最开始怎么想（惯性）

我会开：

  • rows[m] 记录哪些行要清零
  • cols[n] 记录哪些列要清零 这是 O(m+n) 空间，不满足进阶要求。

最优思路一句话（同题型但换“存储位置”）

把 rows/cols 标记数组“藏进矩阵本身”：用第一列标记行、第一行标记列；额外用两个布尔保存第一行/列原始是否有 0。

我最大的疑惑：第一行/第一列被覆盖了，怎么恢复？

这里容易产生误解：你以为“需要恢复第一行/列的原值”。 实际上不需要恢复，因为第一行/第一列最终是否要清零，只取决于它们原始是否含 0，而这份信息我们在覆盖前用两个布尔保存了：

  • first_row_zero
  • first_col_zero

覆盖阶段写入的 0 是“标记”，而这些标记对应的行/列最终本来就要清零，因此不会出现“写错了还得恢复”的情况。

惯性 → 最优转换（机械步骤：四阶段固定）

  1. 扫第一列/第一行 → 保存 first_col_zero / first_row_zero
  2. 扫内部 → 遇 0 就写标记到 matrix[i][0] / matrix[0][j]
  3. 再扫内部 → 根据标记清零内部
  4. 最后根据两个布尔 → 清第一行/第一列

8) 边界框遍历：LeetCode 54 — 螺旋矩阵

题意

按顺时针螺旋顺序返回矩阵所有元素。

背景：我为什么写不出循环？

我卡的不是“理解螺旋”，而是：

  • 不知道循环结构怎么组织
  • 不知道边界条件怎么写才不重复遍历

最优思路一句话（与惯性不同：要换“遍历骨架”）

维护闭区间边界框 [top,bottom]×[left,right]；每轮固定走上右下左四条边并收缩；下/左前加两条 if 防止单行/单列重复。

我当时容易写错的点（逻辑链补全）

  • while 条件应该是“还有未遍历区域”：top<=bottom and left<=right

  • 下边/左边会在只剩一行或一列时造成重复，所以需要：

    • if top <= bottom: 才走下边
    • if left <= right: 才走左边

口诀：

闭区间框住未遍历区域；每轮上右下左走一圈并收缩；下/左两步用 top<=bottom、left<=right 防重复。

最简代码模板

9) 坐标映射：LeetCode 48 — 旋转图像（顺时针 90°）

题意

给 n×n 矩阵，原地顺时针旋转 90°。

背景：我最开始怎么想（惯性）

“从外环到内环做四点交换”，但我卡在：

  • 对应位置如何写
  • 容易写错坐标

最优思路一句话（换实现方式，但不换问题本质）

用坐标映射 (i,j)→(j,n-1-i)，最稳实现是：先转置，再每行反转。

为什么稳？

  • 你不需要手写四点交换的坐标
  • 转置 + 反转是固定模板，错误率低

结尾：以后我怎么用这套方法

我现在的刷题流程是：

  1. 先写“能跑的惯性版本”
  2. 问自己：这题属于哪一类状态变量？（last_index / diff+nonzero / need+missing / running_end / prefix-suffix / reuse-marker）
  3. 按该类的固定模板做重构，把“最优”变成“可机械转换”

我的目标不是每次一次写对，而是让“最优代码结构”成为下一次的惯性。
