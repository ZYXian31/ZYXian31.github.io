题目一：【力扣347. 前 K 个高频元素】

基础信息：中等 | 标签：数组、哈希表、桶排序、堆（优先队列） | 复盘日期：2026-03-12

力扣原题描述

给你一个整数数组 nums 和一个整数 k ，请你返回其中出现频率前 k 高的元素。你可以按 任意顺序 返回答案。

示例 1： 输入：nums = [1,1,1,2,2,3], k = 2 输出：[1,2]

示例 2： 输入：nums = [1], k = 1 输出：[1]

提示：

  • 1 <= nums.length <= 10^5
  • -10^4 <= nums[i] <= 10^4
  • k 的取值范围是 [1, 数组中不相同的元素的个数]
  • 题目数据保证答案唯一

进阶： 你所设计算法的时间复杂度 必须 优于 O(n log n) ，其中 n 是数组大小。

一、我的初始思路 & 认知存在的核心问题

  1. 思路层面误区：成功想到用哈希表建立「数字→频次」的映射，但后续陷入「必须对频次进行排序才能选前k高」的定式思维；考虑过堆排序，但对堆的复杂度优化边界和桶排序的适用场景认知模糊，未找到突破O(n log n)的方法。
  2. 编码层面踩坑：
    • 致命bug：循环变量名k覆盖了函数参数k（前k个元素的k），导致后续逻辑完全错误；
    • 逻辑bug：使用extend批量加入桶中元素，可能返回超过k个元素；
    • 代码习惯：引入冗余变量ec统计答案数量，未直接利用len(ans)。
  3. 认知层面盲区：未意识到「元素出现频次的范围是有限的（1到n）」这一关键特性，不知道可以利用该特性通过桶排序避免比较排序。

二、惯性思路 → 最优解的思维转换方法

  1. 破局关键：前k高频元素的核心不是「对频次进行排序」，而是「利用频次范围有限（1~n）的特性直接定位高频元素」，彻底规避比较排序的时间开销。
  2. 思维转换动作：从「统计频次后排序/堆选」，转换为「统计频次后按频次分桶，倒序遍历桶直接取高频」，用空间换时间将复杂度从O(n log n)降至O(n)。
  3. 可复用经验：对于「特征范围有限的问题（如频次、年龄、分数段）」，优先考虑桶排序，利用特征的范围特性避免比较排序；遇到「前k大/小」问题，除了堆，还可以思考是否有可利用的特征范围。

三、最优解法实现

  • 编程语言：Python3
  • 代码实现
  • 时间复杂度分析：O(n)
    • 统计频次：O(n)
    • 元素入桶：O(m)，m为不同元素个数（m ≤ n）
    • 倒序遍历桶：O(n)
    • 总复杂度：O(n + m + n) = O(n)
  • 空间复杂度分析：O(n)
    • 哈希表：O(m)
    • 桶数组：O(n)
    • 总空间：O(n)

四、核心思路 一句话极简总结

用哈希表统计元素频次，利用频次范围1~n的特性建立「频次-元素」桶数组，倒序遍历桶直接取前k个高频元素，彻底规避比较排序。

补充知识点

  1. 桶排序核心思想：将数据按特征分到有限数量的桶里，每个桶内再单独排序（或直接利用桶的顺序性），利用桶的范围特性规避整体比较排序。

题目二：【力扣295. 数据流的中位数】

基础信息：困难 | 标签：设计、堆（优先队列）、数据流 | 复盘日期：2026-03-12

力扣原题描述

中位数是有序整数列表中的中间值。如果列表的大小是偶数，则没有中间值，中位数是两个中间值的平均值。

例如 arr = [2,3,4] 的中位数是 3 。 例如 arr = [2,3] 的中位数是 (2 + 3) / 2 = 2.5 。

实现 MedianFinder 类:

  • MedianFinder() 初始化 MedianFinder 对象。
  • void addNum(int num) 将数据流中的整数 num 添加到数据结构中。
  • double findMedian() 返回到目前为止所有元素的中位数。与实际答案相差 10^-5 以内的答案将被接受。

示例 1： 输入 ["MedianFinder", "addNum", "addNum", "findMedian", "addNum", "findMedian"] [[], [1], [2], [], [3], []] 输出 [null, null, null, 1.5, null, 2.0]

提示：

  • -10^5 <= num <= 10^5
  • 在调用 findMedian 之前，数据结构中至少有一个元素
  • 最多 5 * 10^4 次调用 addNum 和 findMedian

一、我的初始思路 & 认知存在的核心问题

  1. 思路层面误区：完全无思路，误以为题目要求 addNum 和 findMedian 都必须是 O(1)，陷入“无法同时满足快速插入和快速取中位数”的困境；未意识到中位数只依赖“左半边最大值”和“右半边最小值”，而非全局完整排序。
  2. 认知层面盲区：不知道可以用两个堆配合来分别维护左右两半数据，对“大根堆+小根堆”的组合使用场景不熟悉；对 Python 中如何模拟大根堆（存相反数）缺乏了解。

二、惯性思路 → 最优解的思维转换方法

  1. 破局关键：中位数的核心不是“全局有序”，而是“快速获取左右两半的边界值”——左半边最大值和右半边最小值。
  2. 思维转换动作：从“试图用单个数据结构同时满足快速插入和快速取中位数”，转换为“用两个堆分工：大根堆存较小一半（快速取左边界），小根堆存较大一半（快速取右边界）”，并通过插入后的平衡操作维持两个堆的大小差不超过1。
  3. 可复用经验：遇到“需要同时获取最大值和最小值”或“需要维护数据的中间状态”的问题，优先考虑“双堆/双栈/双队列”的分工配合；先明确目标查询依赖的最小信息集，再选择对应的数据结构。

三、最优解法实现

  • 编程语言：Python3
  • 代码实现
  • 时间复杂度分析：
    • addNum()：涉及常数次堆插入/弹出，每次堆操作为 O(log n)，总复杂度 O(log n)
    • findMedian()：直接访问堆顶，O(1)
  • 空间复杂度分析：存储所有元素，O(n)

四、核心思路 一句话极简总结

用大根堆维护较小一半元素、小根堆维护较大一半元素并保持大小平衡，中位数直接由两个堆顶得到。

补充知识点

  1. 双堆分工的核心逻辑：
    • 大根堆（small）：存左半边，快速取最大值
    • 小根堆（large）：存右半边，快速取最小值

题目三：【力扣55. 跳跃游戏】

基础信息：中等 | 标签：贪心、数组、动态规划 | 复盘日期：2026-03-12

力扣原题描述

给你一个非负整数数组 nums ，你最初位于数组的 第一个下标 。数组中的每个元素代表你在该位置可以跳跃的最大长度。

判断你是否能够到达最后一个下标。

示例 1： 输入：nums = [2,3,1,1,4] 输出：true 解释：可以先跳 1 步，从下标 0 到达下标 1, 然后再从下标 1 跳 3 步到达最后一个下标。

示例 2： 输入：nums = [3,2,1,0,4] 输出：false 解释：无论怎样，总会到达下标为 3 的位置。但该下标的最大跳跃长度是 0 ， 所以永远不可能到达最后一个下标。

提示：

  • 1 <= nums.length <= 3 * 10^4
  • 0 <= nums[i] <= 10^5

一、我的初始思路 & 认知存在的核心问题

  1. 思路层面误区：采用“局部选点”的贪心策略——在当前位置能跳到的范围内，选一个“下一步之后还能跳得最远”的位置，试图构造具体跳跃路径；但题目只问“能不能到达”，不需要知道具体路径，思路绕远且容易出错。
  2. 编码层面踩坑：
    • 致命bug：更新条件用 > 而非 >=，当多个候选位置的“未来最远覆盖”相同时，pre_max_idx 不更新，导致 i 停在原地进入死循环；
    • 边界bug：未单独处理 n == 1 的情况，导致 [0] 这种“起点即终点”的 case 被误判为 False；
    • 冗余逻辑：引入 pre_max_idx 和 next_loc 试图选点，而非直接维护全局范围。
  3. 认知层面盲区：未意识到“覆盖范围”思维比“具体路径”思维更适合此类问题，不知道可以通过维护一个全局的 farthest 变量来简化问题。

二、惯性思路 → 最优解的思维转换方法

  1. 破局关键：跳跃游戏的核心不是“每一步跳哪”，而是“当前所有可达位置合起来能把最远边界推进到哪里”。
  2. 思维转换动作：从“局部选点构造路径”，转换为“全局维护最远可达范围”；从“关心下一步落点”，转换为“关心当前位置是否可达、以及能否用它扩展边界”。
  3. 可复用经验：对于“可达性”“覆盖范围”类问题，优先考虑“维护全局边界”的贪心策略，而非“构造具体路径”；先明确题目是否真的需要路径，若不需要，尽量简化为状态/范围的维护。

三、最优解法实现

  • 编程语言：Python3
  • 代码实现
  • 时间复杂度分析：仅需遍历数组一次，O(n)
  • 空间复杂度分析：仅用常数个变量，O(1)

四、核心思路 一句话极简总结

维护当前最远可达位置，遍历过程中若未遇到不可达位置且能覆盖终点则成功。

题目四： 【力扣45. 跳跃游戏 II】

基础信息：中等 | 标签：贪心、数组、动态规划 | 复盘日期：2026-03-12

力扣原题描述

给定一个长度为 n 的 0 索引整数数组 nums。初始位置为 nums[0]。

每个元素 nums[i] 表示从索引 i 向前跳转的最大长度。换句话说，如果你在 nums[i] 处，你可以跳转到任意 nums[i + j] 处:

  • 0 <= j <= nums[i]
  • i + j < n

返回到达 nums[n - 1] 的最小跳跃次数。生成的测试用例可以到达 nums[n - 1]。

示例 1： 输入: nums = [2,3,1,1,4] 输出: 2 解释: 跳到最后一个位置的最小跳跃数是 2。从下标 0 跳到下标 1，跳 1 步，然后跳 3 步到达数组的最后一个位置。

示例 2： 输入: nums = [2,3,0,1,4] 输出: 2

提示：

  • 1 <= nums.length <= 10^4
  • 0 <= nums[i] <= 1000
  • 题目保证可以到达 nums[n-1]

一、我的初始思路 & 认知存在的核心问题

  1. 第一次踩坑：按点贪心，混淆信息收集与决策

    • 思路层面误区：采用“按点贪心”策略——遍历过程中只要发现“当前位置能扩展最远覆盖范围”，就直接将跳跃次数 jump_num 加 1；混淆了“更新下一跳候选信息”和“实际完成一次跳跃”，导致跳数增加时机完全错误。
    • 编码层面踩坑：
      • 致命bug：仅用一个变量 max_interval 同时承担“当前层边界”和“下一层候选边界”，逻辑混乱；
      • 时机错误：将“更新最远覆盖”等同于“多跳一步”，在同一层范围内多次更新候选时会错误累加跳数。
    • 认知层面盲区：未意识到这题是“按层推进”的贪心（类似 BFS 层级遍历），不知道需要区分“当前跳跃的边界 end”和“下一层最远覆盖 farthest”两个变量；对“答案加一的时机”缺乏清晰的不变量定义。

  2. 第二次踩坑：掌握框架后，颠倒信息收集与决策的顺序

    • 思路层面误区：已经掌握了「按层推进」的核心框架，知道要维护 farthest、end、steps 三个状态，但完全搞反了「收集信息」和「执行决策」的顺序。
    • 编码层面踩坑：
      • 致命bug：在 i == end 时先执行「跳步+更新边界」，再用当前位置 i 更新 farthest，漏掉了当前层边界点 i 的跳跃能力贡献；
      • 结果偏差：边界点的信息缺失会导致 end 无法及时更新到正确位置，最终跳跃次数统计偏小。
    • 认知层面盲区：对「i == end 的含义」理解不够深刻——i == end 表示「当前层的最后一个点正是 i」，这个点必须先参与下一层边界的信息收集，才能触发跳步决策。

二、惯性思路 → 最优解的思维转换方法

  1. 破局关键：
    • 从“按点决策”升级为“按层决策”：最少跳跃次数的核心不是“每发现一个更优落点就跳一步”，而是“在当前跳跃能覆盖的整层范围内，收集完所有下一跳候选信息后，才真正跳一次”；
    • 严格遵循“先收集信息，后执行决策”的顺序：i == end 时，必须先确保当前层所有点（包括边界点 i）都已贡献下一层边界信息，再触发跳步。
  2. 思维转换动作：
    • 从“按点决策（遍历到一个点就可能跳一步）”，转换为“按层决策（扫完当前层边界 end 才跳一步）”；
    • 从“单个变量维护最远覆盖”，转换为“双变量分工：end 维护当前层边界，farthest 维护下一层最远覆盖”；

基础信息：中等 | 标签：数组、动态规划、贪心、二分查找 | 复盘日期：2026-03-12

力扣原题描述

给你一个整数数组 nums ，找到其中最长严格递增子序列的长度。

子序列 是由数组派生而来的序列，删除（或不删除）数组中的元素而不改变其余元素的顺序。例如，[3,6,2,7] 是数组 [0,3,1,6,2,2,7] 的子序列。

示例 1： 输入：nums = [10,9,2,5,3,7,101,18] 输出：4 解释：最长递增子序列是 [2,3,7,101]，因此长度为 4 。

示例 2： 输入：nums = [0,1,0,3,2,3] 输出：4

示例 3： 输入：nums = [7,7,7,7,7,7,7] 输出：1

提示：

  • 1 <= nums.length <= 2500
  • -10^4 <= nums[i] <= 10^4

进阶： 你能将算法的时间复杂度降低到 O(n log(n)) 吗?

一、我的初始思路 & 认知存在的核心问题

  1. 第一次踩坑：混淆「子序列」与「连续子数组」

    • 思路层面误区：
      • 状态定义方向是对的：想维护「以当前位置为尾的最长严格递增子序列长度」；
      • 但转移逻辑完全错了：只比较 nums[i] 和 nums[i-1]，一旦 nums[i] <= nums[i-1] 就将状态重置为 1，本质上做成了最长连续递增子数组，而非子序列。
    • 编码层面踩坑：
      • 致命bug：转移逻辑仅依赖前一个位置 i-1，而非所有 j < i 的位置；
      • 结果偏差：例如 nums = [10,9,2,5,3,7,101,18]，代码会得到 3，但正确答案是 4（因为可以跳过 5 直接接 3）。
    • 认知层面盲区：
      • 对「子序列」和「子数组」的核心区别不敏感：子序列可以跳着选，不要求连续；子数组必须连续；
      • 不知道「以 i 为结尾的 LIS」可以接在任何一个 j < i 且 nums[j] < nums[i] 的位置后面，而非只能接在 i-1 后面。

  2. 对 O(n log n) 优化的认知空白：

    • 不知道可以通过维护「每个长度的递增子序列的最小结尾」来将复杂度从 O(n²) 降到 O(n log n)；
    • 对 tails 数组的含义（长度为 k+1 的递增子序列的最小结尾）和为什么它是递增的、可以二分，缺乏理解。

二、惯性思路 → 最优解的思维转换方法

  1. 从 O(n) 滚动变量到 O(n²) 动态规划：

    • 破局关键：明确「子序列可以不连续」，因此「以 i 为结尾的 LIS」不一定接在 i-1 后面，而要从所有 j < i 且 nums[j] < nums[i] 的位置中选最优的。
    • 思维转换动作：
      • 从「只用一个滚动变量 pre」，转换为「用 dp 数组存储每个位置的状态」；
      • 从「只比较 nums[i] 和 nums[i-1]」，转换为「枚举所有 j < i，找 nums[j] < nums[i] 中 dp[j] 最大的那个」。

  2. 从 O(n²) DP 到 O(n log n) 贪心 + 二分：

    • 破局关键：不再关心「每个位置结尾的 LIS 长度」，而是关心「对于每个固定长度的递增子序列，它的结尾最小能是多少」（结尾越小，后面越容易接更大的数）。
    • 思维转换动作：
      • 定义 tails[k]：长度为 k+1 的递增子序列中，结尾元素的最小可能值；
      • 遍历每个数 num：
        • 若 num 比 tails 最后一个元素大，直接追加（找到了更长的递增子序列）；
        • 否则，用二分找到 tails 中第一个 >= num 的位置，替换成 num（保持长度不变，但让结尾更小，更有潜力）。

三、最优解法实现

  • 编程语言：Python3
  • 解法1：O(n²) 动态规划（直观，基础）

  • 时间复杂度：O(n²)（外层 n，内层 n）

  • 空间复杂度：O(n)

  • 解法2：O(n log n) 贪心 + 二分查找（最优，进阶）

  • 时间复杂度：O(n log n)（遍历 n 个数，每次二分 O(log n)）
  • 空间复杂度：O(n)

四、核心思路 一句话极简总结

O(n²) 版本：定义 dp[i] 为以 nums[i] 结尾的 LIS 长度，枚举所有 j < i 且 nums[j] < nums[i] 的位置转移，取最大值。 O(n log n) 版本：维护 tails 数组表示「每个长度的递增子序列的最小结尾」，遍历每个数用二分找到其应在的位置，追加或替换，最终 tails 的长度即为答案。

  • 跳跃游戏I：维护全局最远可达位置，判断能不能到终点；
  • 买卖股票的最佳时机：维护历史最低价格和当前最大利润，求最优收益；
  • 最长无重复子串：维护滑动窗口边界和字符最后出现位置，求最长合法子串。

遍历顺序完全一致，但核心逻辑完全不同。决定代码能不能跑对的，从来不是for循环怎么写，而是循环里你维护了什么、怎么更新。

算法题的核心四要素

所有算法题，本质上都可以拆解为四个核心问题。把这四个问题想清楚，代码自然就能写对，不会出现逻辑混乱、边界bug、时机错误的问题。

1. 定义状态：我到底要维护什么信息？

这是写代码前最核心的问题，也是所有逻辑的起点。你要先明确，为了算出最终答案，需要全程跟踪哪些数据。

  • 跳跃游戏II：需要维护「已跳跃次数steps」、「当前跳跃的边界end」、「下一跳的最远覆盖farthest」三个核心状态；
  • 数据流的中位数：需要维护「存较小一半的大根堆small」、「存较大一半的小根堆large」两个核心状态；
  • 前K个高频元素：需要维护「数字-频次的哈希映射」、「按频次分桶的桶数组」两个核心状态。

状态定义错了，后续的遍历写得再完美，也只会南辕北辙。

2. 状态转移：新的信息进来，怎么更新状态？

明确了要维护什么，接下来就要定义规则：每遍历到一个新元素、拿到一条新信息，怎么更新你维护的状态？

  • 跳跃游戏II：每遍历到位置i，就要用i + nums[i]去更新下一跳的最远覆盖farthest；
  • 数据流的中位数：每新增一个数字，就要先放入大根堆，再做堆顶元素的转移，保证两个堆的顺序正确。

状态转移的规则，必须严格贴合题目的逻辑，不能想当然。

3. 维护不变量：哪些规则必须全程成立？

不变量，就是从代码开始执行到结束，必须始终保持成立的规则。这是保证代码逻辑不出错的核心，也是很多人写代码时最容易忽略的点。

  • 数据流的中位数：必须全程保证「small堆的所有元素 ≤ large堆的所有元素」、「两个堆的大小差不超过1」，这两个不变量一旦被打破，中位数的计算就会出错；
  • 跳跃游戏II：必须保证「end是当前steps次跳跃能到达的最远边界」，这个不变量是确定跳跃时机的核心；
  • 前K个高频元素：桶数组的下标必须严格对应「元素出现的频次」，否则倒序取数的逻辑就会失效。

写代码的过程，本质上就是在每一步操作后，都维护好你定义的不变量。

4. 明确更新时机：什么时候更新答案？什么时候做决策？

这是我刷题过程中踩坑最多的地方，也是绝大多数bug的来源。很多时候，状态定义对了，转移规则也对了，但就是因为更新时机错了，结果完全不对。

  • 跳跃游戏II的核心bug：我把「更新下一跳最远覆盖farthest（收集信息）」，当成了「完成一次跳跃（执行决策）」，每更新一次farthest就给跳跃次数+1，完全搞错了答案更新的时机；
  • 正确的时机：只有当遍历到当前层的边界end时，才说明当前这一跳的所有候选信息已经收集完毕，必须执行一次跳跃，此时才能给steps+1。

这里的核心认知是：收集信息 ≠ 执行决策。遍历过程中绝大多数操作，都是在为最终的决策收集信息，只有当触发了明确的规则（比如扫完当前层、窗口合法、递归回溯完成），才应该更新答案、执行决策。

可复用的5步读题框架

养成「先想清楚再写代码」的习惯，比刷100道题更重要。以后拿到一道新题，我会先强迫自己回答这5个问题，再动手写代码：

  1. 这道题的最终答案，依赖哪些核心信息？
  2. 这些信息，能不能边遍历边维护？需要定义哪些状态？
  3. 遍历过程中，状态的转移规则是什么？
  4. 必须全程维护的不变量是什么？
  5. 什么时候更新答案？触发决策的条件是什么？

把这5个问题的答案想清楚，再动手写代码，会极大减少bug的出现，也能让你彻底理解题目，而不是背模板、碰运气。

踩坑复盘：我犯过的典型错误

结合这段时间的刷题经历，我发现自己的绝大多数错误，都源于对上面四个核心要素的理解不到位：

  1. 更新时机错误：最典型的就是跳跃游戏II里，把「收集信息」和「执行决策」混为一谈，导致跳跃次数统计错误；
  2. 不变量破坏：写前K个高频元素时，循环变量k覆盖了函数入参k，打破了「入参k是要返回的元素个数」这个核心不变量，导致逻辑完全错误；
  3. 状态定义冗余/偏差：跳跃游戏I的初始代码，非要维护「下一个最优落点」，而不是直接维护「全局最远可达位置」，把简单问题复杂化，还引入了死循环的bug；
  4. 边界处理缺失：没有先判断「起点即终点」的最小case，导致[0]这种用例被误判。

最后总结

写代码的核心，从来不是「把遍历写好」，而是**「把问题抽象成清晰的状态、规则和更新时机」**。

遍历是代码的骨架，但状态、不变量和更新时机，才是算法的灵魂。比起死记硬背题解模板，养成「先想清楚核心四要素，再动手写代码」的习惯，才能真正从「背题做题」变成「理解解题」，也才能把一道题的解法，复用给一类题。

补充：代码题的核心方法论：遍历是载体，更核心的是“状态+不变量+更新时机”

GPT 明确：遍历很重要，但它只是执行状态转移的载体，而非全部。代码题的本质可拆解为以下五个问题：

1. 你要维护什么状态？

比如：

  • 跳跃游戏中的“当前最远可达位置”“当前跳跃边界”；
  • 股票问题中的“当前最低价格”“当前最大利润”；
  • 滑动窗口问题中的“窗口左右边界”“字符最后出现位置”。

2. 状态怎么转移？

比如：

  • 新元素加入时，窗口/最远范围怎么更新；
  • 选/不选当前元素时，DP 值怎么变化。

3. 哪些量必须一直成立（不变量）？

比如：

  • 数据流中位数问题中“左堆所有元素 ≤ 右堆所有元素”；
  • 跳跃游戏 II 中“end 表示当前跳跃的边界”。

4. 什么时候更新答案？

这是最容易出错的点：

  • 是每看到更优候选就更新，还是等一整层扫完再更新？
  • 是递归回来更新，还是窗口合法时更新？

5. 用什么方式访问状态（遍历）？

比如：从左到右、双指针、DFS、BFS、回溯等。

三、读题后的思考框架（训练“会做题”的核心）

遇到题先强迫自己问：

  1. 最终答案依赖哪些信息？
  2. 这些信息能不能边遍历边维护？
  3. 维护时需要几个变量/数组/数据结构？
  4. 什么时候更新答案？
  5. 什么条件下说明失败/成功/需要换层/收缩窗口？

四、「跳跃游戏 II」最优代码详解（结合分层视角）

1. 变量含义

  • steps：已经跳了多少次；
  • end：用 steps 次跳跃，当前最远能到哪（当前层边界）；
  • farthest：在当前层范围内，下一跳最远能到哪（下一层边界）。

2. 核心逻辑：按层推进（类似波纹扩散）

  • 第 1 跳覆盖一层范围，第 2 跳基于前一层扩展到更远的一层；
  • 遍历过程中用 farthest 收集“下一层最远边界”的信息；
  • 只有当扫完当前层边界 end 时，才说明必须真的跳一次（steps += 1，并更新 end = farthest）。

3. 逐行模拟（以 nums = [2,3,1,1,4] 为例）

  • 初始：steps=0, end=0, farthest=0；
  • i=0：更新 farthest=2，因 i==end，steps=1，end=2（第 1 跳覆盖到 2）；
  • i=1：更新 farthest=4，未到 end，不跳；
  • i=2：farthest 保持 4，因 i==end，steps=2，end=4（第 2 跳覆盖到终点）。

4. 与错误代码的核心区别

  • 错误代码：把“更新下一跳候选覆盖”误当成“实际完成一次跳跃”，每更新一次 farthest 就加一次跳数；
  • 正确代码：扫完当前层边界 end 才加跳数，严格区分“收集候选信息”和“实际执行操作”。

五、一句话总结

写程序的核心不是“把遍历写好”，而是**“把问题抽象成一组状态、规则和更新时机”**；「跳跃游戏 II」的本质是用 end 维护当前层边界、farthest 统计下一层边界，扫完当前层才跳一次。
