题目一：二分查找（LeetCode 35. 搜索插入位置）

题目基础信息

  • 题目链接：LeetCode 35. 搜索插入位置
  • 题目难度：简单
  • 核心考点：有序数组的二分查找、目标值插入位置定位、区间不变性把控
  • 题目简述：给定一个升序排列的整数数组和一个目标值，在数组中找到目标值并返回其索引；如果目标值不存在于数组中，返回它按顺序插入的位置。算法时间复杂度必须为 O(log n)。

我的初始思路与核心痛点

最初写这道题，我虽然知道要用二分查找，但全程是「凭模糊记忆堆代码」，核心问题非常突出：

  1. 指针初始化纠结：不确定右边界r该写len(nums)还是len(nums)-1，全靠用例试错
  2. 循环条件混乱：偶尔把l <= r写成l < r，完全不知道两者的适用场景
  3. 指针更新无依据：mid不满足条件时，分不清该写r = mid还是r = mid - 1
  4. 返回值全靠蒙：循环结束后，不知道该返回l还是r，只能靠测试用例反推

本质上，我只记住了二分查找的代码壳子，完全没理解背后的核心逻辑，更没有形成可复用、可推导的解题范式，换一道变种题就会再次踩坑。

核心认知突破：二分的本质是「区间定义驱动全流程逻辑」

这道题给我最大的收获，不是解出了题目本身，而是彻底搞懂了：二分查找的所有代码细节，都由你定义的「搜索区间」决定，全程必须坚守「区间不变性」。

二分查找的核心，就是不断缩小搜索区间，直到找到目标值，或确定目标值的插入位置。而所有代码细节，都要和你最初定义的区间规则完全匹配，一旦规则不统一，必然会出现死循环、越界、结果错误等问题。

两种核心区间定义的全链路规则

区间类型 初始化规则 循环条件 核心不变性 循环结束状态
左闭右闭 [l, r] l=0, r=len(nums)-1 l <= r 目标值始终在闭区间[l, r]内，两个端点都可能是答案 l > r，区间变为空集，r和l左右错位
左闭右开 [l, r) l=0, r=len(nums) l < r 目标值始终在左闭右开区间[l, r)内，左端点可能是答案，右端点不纳入搜索范围 l == r，区间收缩为一个点，该点即为目标位置

思路迁移范式：从「凭感觉写」到「按步骤推导」

通过这道题，我固化了一套通用的二分查找解题四步范式，以后遇到任何二分题，都能按这个流程，从区间定义推导出全部代码，彻底告别死记硬背和试错。

范式四步走（100%适配所有二分基础题）

  1. 先定区间规则：优先选择左闭右开[l, r)（官方推荐，不易越界，适配场景更广），明确本次搜索的区间边界规则。
  2. 匹配循环条件：区间定义直接决定循环条件——闭区间用l <= r，开区间用l < r，确保循环结束时区间状态符合预期。
  3. 推导指针更新规则：全程坚守区间不变性，排除的元素绝不能再出现在后续搜索区间里：
    • 若nums[mid] < target：说明目标值在mid右侧，mid已被排除，两种区间都统一写l = mid + 1
    • 若nums[mid] > target：闭区间中mid已被排除，写r = mid - 1；开区间中mid仍在右边界范围内，写r = mid
  4. 确定最终返回值：两种区间规则下，循环结束后统一返回l，无需纠结l和r的选择：
    • 左闭右闭：循环结束时nums[r] < target < nums[l]，l就是插入位置
    • 左闭右开：循环结束时l == r，这个点就是目标值的位置/插入位置

面试高频避坑指南（刷题踩过的坑，提前规避）

  1. 规则不统一是最大的坑：最常见的错误，就是初始化用了左闭右闭的r = len(nums)-1，指针更新却写了r = mid，直接导致死循环或结果错误，必须全程坚守区间不变性。
  2. mid计算溢出问题：Python中整数无溢出风险，但Java、C++等语言，必须用l + (r - l) // 2替代(l + r) // 2，避免两数相加超出整数范围。
  3. 忽略二分的前提条件：二分查找的核心前提是「数组有序（或部分有序可判断区间取舍）」，不能对无序数组无脑套用二分模板。
  4. 不要死记返回值：一定要理解循环结束时的区间状态，而不是死记「返回l还是返回r」，换个变种场景就会出错。

  2. 整理二分变种题（旋转有序数组、找峰值、二分答案类题目）的范式延伸，把这套区间逻辑迁移到更复杂的场景中。
  3. 把上述模板做成速记卡，面试前10分钟快速过一遍，彻底解决二分的边界焦虑，确保基础题不丢分。

二分查找是面试的基础必考题，看似代码简单，实则边界细节拉满。只有吃透「区间不变性」这个核心，才能从「碰运气写对」变成「100%一次写对」，把送分题稳稳拿住。

题目二：在排序数组中查找元素的第一个和最后一个位置（LeetCode 34）

题目基础信息

  • 题目链接：LeetCode 34. 在排序数组中查找元素的第一个和最后一个位置
  • 题目难度：中等
  • 核心考点：有序数组二分左右边界查找、二分模板一致性、数组越界防护
  • 题目简述：给定一个按照升序排列的整数数组 nums，和一个目标值 target。找出给定目标值在数组中的开始位置和结束位置。如果数组中不存在目标值，返回 [-1, -1]，必须设计并实现时间复杂度为 O(log n) 的算法。

我的初始思路与核心痛点

拿到题目，我的惯性思路是「拆分问题，分别实现」：既然要找左、右两个边界，那就写两段二分代码，第一段找左边界，第二段找右边界。但最终写出来的代码不仅有多处致命错误，还完全违背了之前固化的「区间定义驱动逻辑」的原则，核心问题集中在3点：

  1. 左边界越界检查缺失：左边界用了左闭右开[l, r)的模板，循环结束后直接访问nums[r]，但当target大于数组所有元素时，最终l=r=len(nums)，直接触发数组越界报错。
  2. 右边界代码触发死循环：为了找右边界，我临时拼凑了一套不伦不类的模板，用了向下取整的mid=(l+r)//2，满足条件时却写了l=mid，当区间收缩到l和r相邻时，mid会始终等于l，区间无法继续收缩，直接陷入无限循环。
  3. 逻辑混乱导致结果错误：变量命名与实际含义完全相反，把左边界存成了rb，最后返回时把右边界、左边界顺序写反，哪怕代码修好了边界问题，结果也完全错误。

本质上，我陷入了「为不同场景硬套不同模板」的误区，之前记住的左闭右开区间规则，在写右边界时完全抛在了脑后，既没有坚守区间不变性，也没有找到统一的解题逻辑，越写越乱，全靠试错改代码。

核心认知突破：用「一个核心模板+数学转换」，替代多套零散模板

这道题给我最大的收获，不是修好了代码里的bug，而是彻底打破了「不同边界要写不同二分代码」的惯性思维，建立了**「一模板通吃所有二分基础场景」**的核心认知：

只需要记住lower_bound（找第一个>= target的位置）这一个模板，就能通过简单的数学转换，同时解决左边界、右边界、插入位置、精准查找等所有基础二分场景。

针对本题的核心转换逻辑：

  • 左边界 = lower_bound(target)（天然就是第一个>= target的位置，完全匹配需求）
  • 右边界 = lower_bound(target + 1) - 1（第一个>= target+1的位置，就是第一个大于target的位置，往前推一位就是最后一个等于target的位置）

这个思路直接把「写两套容易出错的二分代码」，简化成了「两次调用同一个固定模板」，彻底规避了边界规则不统一、死循环、越界等问题。

思路迁移范式：从「拼凑多套模板」到「统一逻辑复用」

通过这道题，我彻底完成了二分解题思路的范式升级，从原来的「场景化硬套代码」，转变为「核心逻辑复用」，具体迁移路径如下：

我的旧思路（错误、不可靠）

  1. 拆分问题：左边界要找第一个>= target，右边界要找最后一个<= target
  2. 为两个边界分别设计两套二分代码，分别定义区间、循环条件、指针更新规则
  3. 分别处理两套代码的边界检查、返回值，最后拼接结果 → 核心问题：两套代码规则不统一，极易写错，调试成本极高，面试时很容易慌神出错。

最优解新思路（正确、可复用、零出错）

  1. 固化核心逻辑：只写一次lower_bound模板，严格坚守左闭右开[l, r)的区间规则，永远不变
  2. 数学转换需求：把左右边界的需求，都转换成对lower_bound的调用
  3. 极简结果校验：只需要一次越界和存在性检查，就能完成结果校验 → 核心优势：模板完全固定，只需要记一套规则，没有任何需要临场发挥的内容，面试时100%稳定输出。

模板的通用复用能力

面试高频避坑指南（踩过的坑，绝不再犯）

  1. 越界检查的顺序不能反：必须先判断left == len(nums)，再判断nums[left] != target，否则当left越界时，访问数组会直接报错。
  2. 永远不要为右边界单独写二分代码：只要用lower_bound(target+1)-1的写法，就能100%规避死循环、边界规则混乱的问题，没有任何例外。
  3. 模板规则必须全程坚守：lower_bound的区间、循环条件、指针更新规则，一个字都不要改，不要临场修改模板，避免出现规则不统一的错误。
  4. 变量命名必须语义清晰：左边界就叫left，右边界就叫right，不要用容易混淆的缩写，避免出现返回顺序颠倒的低级错误。

  2. 用统一模板重刷二分基础题：用这个模板重新做LeetCode 35、704、34这三道核心题，彻底固化「模板+转换」的解题思路，不再走回头路。
  3. 拓展模板的变种场景：整理如何用这个模板解决旋转有序数组、二分答案等进阶场景，把这套统一逻辑迁移到更复杂的二分题中。

题目三：搜索旋转排序数组（LeetCode 33）

题目基础信息

  • 题目链接：LeetCode 33. 搜索旋转排序数组
  • 题目难度：中等
  • 核心考点：旋转有序数组的二分查找、有序区间判断、边界退化处理
  • 题目简述：整数数组 nums 按升序排列且值互不相同，在某个下标 k 上进行了旋转（例如 [0,1,2,4,5,6,7] 变为 [4,5,6,7,0,1,2]）。给定旋转后的数组和目标值 target，返回 target 的下标，不存在则返回 -1，必须设计时间复杂度为 O(log n) 的算法。

我的初始思路与核心痛点

一开始拿到这题完全没思路，不知道旋转后的数组怎么用二分。后来知道了「至少一半有序」的核心，但自己写代码时还是踩了两个致命的边界坑：

  1. 有序区间判断漏了单元素情况：用 nums[l] < nums[mid] 判断左边有序，没考虑当 l == mid 时（区间只剩1个元素），单元素也算有序，导致判断逻辑在边界收缩时失效。
  2. 区间判断重复包含已排除的 mid：前面已经单独判断过 nums[mid] == target，但后面写有序区间判断时还是用了 <= nums[mid] 或 >= nums[mid]，逻辑不干净，虽然偶尔能过但容易出错。

本质上，我对「二分边界退化场景」的理解完全缺失，只记住了「一半有序」的壳子，没搞清楚判断条件的细节，导致代码在边界情况（如 nums = [3,1]）下直接出错。

核心认知突破：记住「两个固定判断 + 一个边界意识」，直接套模板

这道题的最优解不需要复杂思考，只要记住三个核心点，就能直接写出无错代码：

  1. 核心前提：旋转后的数组，任意二分后至少有一半是有序的（单元素区间也算有序）。
  2. 固定判断顺序：先判断 nums[mid] == target（找到直接返回），再判断哪边有序，最后判断 target 在不在这个有序区间里。
  3. 边界意识：判断有序用 <=（包含单元素），判断区间用 </>（排除已检查过的 mid）。

思路迁移范式：从「凭感觉写判断」到「三步固定流程」

我的旧思路（错误、不稳定）

  1. 知道要用二分，但不确定怎么处理旋转
  2. 凭感觉写 nums[l] < nums[mid] 判断有序
  3. 区间判断随便写 <=，没考虑 mid 已经被排除 → 核心问题：边界情况必错，调试全靠试错。

最优解新思路（固定、零出错）

严格按照三步流程走，不需要临场发挥：

  1. 第一步：判 mid：如果 nums[mid] == target，直接返回 mid。
  2. 第二步：判哪边有序：用 nums[l] <= nums[mid] 判断左边是否有序（包含单元素），是则处理左边，否则处理右边。
  3. 第三步：判 target 在不在有序区间：
    • 左边有序：判断 nums[l] <= target < nums[mid]，在则缩右边界，否则缩左边界。
    • 右边有序：判断 nums[mid] < target <= nums[r]，在则缩左边界，否则缩右边界。 → 核心优势：流程完全固定，没有任何需要临场思考的内容，边界情况100%稳定。

面试&复习专用：直接背这个模板和记忆口诀

唯一需要背的代码模板

超简单记忆口诀（三句话）

  1. 先看 mid 对不对，对了直接回
  2. 左边有序用 <=，单元素也算
  3. 区间判断排除 mid，左闭右开/右闭左开记清楚

面试高频避坑指南（踩过的坑，绝不再犯）

  1. 有序判断必须用 <=：哪怕数组元素互不相同，也要用 nums[l] <= nums[mid]，因为当 l == mid 时，单元素区间必须被视为有序。
  2. 区间判断必须排除 mid：前面已经单独判断过 nums[mid] == target，所以左边有序时写 target < nums[mid]，右边有序时写 nums[mid] < target，不要重复包含 mid。
  3. 循环条件用 l <= r：这题用左闭右闭区间 [l, r] 最直观，循环条件必须匹配，不要和左闭右开搞混。
  4. 不要尝试找旋转点再二分：直接用「一半有序」的思路，比先找旋转点再两次二分更简单、更不容易出错。

题目五：寻找两个正序数组的中位数（LeetCode 4）

题目基础信息

  • 题目链接：LeetCode 4. 寻找两个正序数组的中位数
  • 题目难度：困难
  • 核心考点：双有序数组的二分切分、中位数本质转换、边界越界处理、单调性空间利用
  • 题目简述：给定两个大小分别为 m 和 n 的正序（从小到大）数组 nums1 和 nums2，请找出并返回这两个正序数组的中位数，算法时间复杂度必须为 O(log (m+n))。

我的初始思路与核心痛点

一开始完全没思路，第一反应是「合并两个有序数组再直接取中位数」，但这个方法复杂度是 O(m+n)，不符合题目要求。核心痛点集中在3点：

  1. 不知道怎么不合并数组找中位数：对中位数的理解只停留在「合并后中间的数」，没挖到更本质的逻辑。
  2. 对 O(log(m+n)) 没方向：不知道怎么在两个独立的有序数组上用二分，找不到单调性在哪里。
  3. 越界和边界处理完全没概念：想象不到两个数组切分后会有什么边界情况，不知道怎么处理。

本质上，我既没完成「中位数→切分」的本质转换，也没找到这题的单调性空间，完全被「两个数组」的形式吓住了。

核心认知突破：记住「一个本质转换 + 四个固定步骤」，直接套模板

这题虽然是困难题，但只要记住四个核心认知点，就能把它拆解成机械的步骤：

  1. 本质转换：中位数不是「合并后的中间数」，而是「找到一个切分，把所有数分成左右两半，左半部分总数固定，且左半部分所有元素 ≤ 右半部分所有元素」。
  2. 左边总数固定：左半部分总个数统一为 (m + n + 1) // 2，奇数时左边多1个，偶数时左右一样多，一套公式覆盖两种情况。
  3. 只二分短数组：为了避免被动算出的 j 越界，只在更短的数组 nums1 上二分切分位置 i，长数组的切分位置 j 由 j = (m + n + 1) // 2 - i 直接推出。
  4. 检查四个边界值：切分后关注四个数：nums1 左最大值、nums1 右最小值、nums2 左最大值、nums2 右最小值，只要满足「nums1左 <= nums2右 且 nums2左 <= nums1右」，切分就正确。

为什么能二分？

因为切分位置 i 具有单调性：

  • 如果 nums1左 > nums2右：说明 nums1 左边切太多了，i 要往左缩。
  • 如果 nums2左 > nums1右：说明 nums1 左边切太少了，i 要往右扩。 这就是单调性，所以可以二分 i。

思路迁移范式：从「想合并数组」到「二分切分位置」

我的旧思路（错误、复杂度不达标）

  1. 合并两个有序数组，得到一个大的有序数组。
  2. 根据总长度的奇偶性，直接取中间的一个或两个数的平均值。 → 核心问题：复杂度是 O(m+n)，不符合题目要求的 O(log(m+n))，且完全没用到二分。

最优解新思路（固定、复杂度达标、可机械执行）

严格按照五步走，不需要任何复杂思考：

  1. 先交换数组：保证 nums1 是更短的那个，避免 j 越界。
  2. 固定左边总数：计算 total_left = (m + n + 1) // 2。
  3. 二分短数组的 i：i 的范围是 0 ~ m（表示 nums1 左边取 i 个元素）。
  4. 推出 j 并取四个边界值：j = total_left - i，用 -inf 和 inf 处理切在数组两端的边界情况。
  5. 检查切分并计算中位数：满足交叉边界条件后，奇数返回左半部分最大值，偶数返回左半最大值和右半最小值的平均值。 → 核心优势：把困难题拆解成了机械步骤，每一步都有固定的写法，不需要临场发挥。

  • 题目简述：设计一个支持 push、pop、top 操作，并能在常数时间内检索到最小元素的栈。实现 MinStack 类：MinStack() 初始化堆栈对象；void push(int val) 将元素推入堆栈；void pop() 删除堆栈顶部的元素；int top() 获取堆栈顶部的元素；int getMin() 获取堆栈中的最小元素。

我的初始思路与核心痛点

一开始我知道普通栈只能保证栈顶操作 O(1)，但 getMin() 做不到，核心痛点集中在3点：

  1. 本能想“查询时再算”：觉得栈就存元素，要最小值时再遍历，完全不符合 O(1) 要求。
  2. 尝试用单个变量存最小值：维护一个 min_val，但 pop() 时会丢失历史最小值——比如弹出当前最小值后，不知道上一个最小值是谁。
  3. 对“单栈存差值”完全没思路：不知道如何在不使用辅助栈的情况下，把历史最小值的信息保存下来。

本质上，我还没建立「更新时维护答案，而不是查询时再计算」的思维，也不知道可以「把辅助信息编码进栈元素本身」。

核心认知突破：记住「一个思维转变 + 一个编码技巧」，推导出差值法

这道题的核心认知不是背代码，而是理解从“普通思路”到“差值法”的推导路径，只要记住两个关键点，就能自己推导出算法：

  1. 思维转变：从「查询时计算」转为「更新时维护」——不要等 getMin() 时再找，而是在每次 push/pop 时，顺手把“当前最小值”的信息存好。
  2. 编码技巧：把历史最小值编码进栈里——不直接存原值，而是存 val - 当前最小值 的差值，用「diff < 0」作为“这里更新了最小值”的信号。

差值法的核心逻辑（自己就能推）

  1. 先想变量存 min_val：发现 pop() 会丢历史，必须把历史信息存下来。
  2. 不想用辅助栈：那就把信息编码进栈元素里——存 diff = val - min_val。
  3. 用 diff 的符号做信号：
    • 如果 diff >= 0：说明 val >= min_val，最小值没变化，原值可以用 min_val + diff 恢复。
    • 如果 diff < 0：说明 val < min_val，这里更新了最小值，pop() 时可以用 min_val - diff 恢复旧最小值。

思路迁移范式：从「普通栈」到「差值法」的四步推导

我的旧思路（走不通）

  1. 用普通栈存元素，getMin() 时遍历 → 复杂度 O(n)，不行。
  2. 加一个变量 min_val 存当前最小值 → pop() 时丢历史，不行。 → 核心问题：没地方存历史最小值，又不想用辅助栈。

最优解新思路（推导式，不是背代码）

严格按照四步推导，自然想到差值法：

  1. 必须存历史：pop() 要恢复旧最小值，历史信息不能丢。
  2. 编码进栈里：不直接存 val，存 diff = val - min_val，把历史信息藏在差值里。
  3. 用符号做信号：diff < 0 就代表“这里更新了最小值”，pop() 时看到负数就知道要恢复旧 min_val。
  4. 恢复原值和旧 min：
    • top()：diff >= 0 就返回 min_val + diff；diff < 0 就直接返回 min_val。
    • pop()：diff < 0 就用 min_val - diff 恢复旧最小值。 → 核心优势：不是死记代码，而是理解推导路径，面试时哪怕忘了也能自己推出来。

  3. diff 小于 0，就是新最小值
  4. pop 见负数，恢复旧最小值

  2. pop() 只有见负数才恢复 min：不是每次 pop 都要改 min_val，只有弹出的 diff < 0 时，才说明弹出的是当时的最小值，需要恢复。

  2. 随机pivot防最坏，太左去右太右去左
  3. 想写稳的用小根堆，堆顶就是第k大

【⚠️ 高频踩坑 重点避坑（面试必看）】

  1. 绝对不要用单调栈：单调栈解决的是“相邻关系/最近边界”问题，和“全局排名”完全无关，这是方向性错误。
  2. 快速选择必须随机选 pivot：如果不随机，遇到有序数组会退化成 O(n²)，随机选 pivot 能保证平均 O(n)。
  3. Lomuto 分区的不变量要理解：i 是下一个 < pivot 应该放的位置，循环结束后要把 pivot 换到 i 的位置。
  4. 小根堆不是大根堆：维护大小为 k 的小根堆，堆里放当前最大的 k 个数，堆顶就是这 k 个数里最小的，也就是全局第 k 大的。

【📌 巩固理解 核心问题解答】

1. 为什么单调栈不行？

单调栈的弹出逻辑代表的是“当前元素和栈顶有某种相邻支配关系”（如最近更大），它会丢掉大量全局信息，无法维护元素的全局排名。而“第k大”需要的是全局顺序，和相邻关系无关，所以单调栈完全不适用。

2. 快速选择和快排的区别是什么？

  • 快排：每次 partition 后，递归左右两边，时间复杂度 O(n log n)。
  • 快速选择：每次 partition 后，只递归目标下标所在的那一边，平均时间复杂度 O(n)（因为每次平均排除一半元素，总操作数是 n + n/2 + n/4 + ... = 2n）。

题目九.1：数组中的第K个最大元素（LeetCode 215）【踩坑复盘与全场景兼容版】

题目基础信息

  • 题目链接：LeetCode 215. 数组中的第K个最大元素
  • 题目难度：中等
  • 核心考点：快速选择（Quickselect）、Lomuto分区坑点、三路划分、重复元素场景优化、循环逻辑适用边界
  • 题目简述：给定整数数组 nums 和整数 k，返回数组排序后的第 k 个最大的元素（不是第 k 个不同的元素），必须设计并实现时间复杂度为 O(n) 的算法。

我的初始思路与核心踩坑历程

我已经理解了这道题的核心是「第k大 = 找下标为 n-k 的元素」，也确定了用快速选择来实现O(n)复杂度，但写代码时连续踩了三个致命坑，从“代码跑不通”到“跑通了超时”，再到“想优化写法反而逻辑出错”，核心痛点集中在3点：

  1. 分区操作的状态同步完全缺失：交换pivot位置后，依然从旧下标取值，导致partition逻辑完全混乱，基础功能都跑不通。
  2. 对二路分区的局限性毫无认知：修正基础错误后，遇到大量重复元素的测试用例直接超时，完全没意识到普通二路分区在重复元素场景下会退化成O(n²)，甚至以为是随机pivot没起作用。
  3. 对循环结构的适用边界完全不清：学会三路划分解决超时问题后，想强行把三路划分的while循环改成二路分区的for循环形式，完全没理解两种循环的索引控制逻辑差异，导致元素漏检、分类错误。

本质上，我只记住了快速选择的框架，却没吃透partition的核心细节，也没预判到算法的边界场景短板，属于“懂了思路，但写出来的代码一踩就坑”。

【⭐ 核心认知突破（补全所有缺失的思维点）】

这道题的复盘，核心不是改对代码，而是补全了快速选择从“能跑”到“稳定高效”的三个关键认知，彻底搞懂了之前踩坑的根源：

1. 基础认知：partition操作的核心是「状态同步」

我之前犯的最低级错误，根源是完全没意识到：交换操作会改变元素的位置，后续所有取值必须跟着新位置走。

  • 我把随机选中的pivot元素交换到了数组最右侧r的位置，但依然从原来的pivot_idx取值，完全忽略了交换后pivot_idx位置的元素已经被替换成了原来的nums[r]。
  • 这个错误也让我养成了固定习惯：只要把pivot换到固定位置，后续就只从这个固定位置取值，绝对不跟着随机下标走。

2. 进阶认知：二路分区的天然短板，随机化也救不了

我之前以为“加了随机选pivot，就能避免最坏情况”，但完全忽略了：随机化只能解决“有序数组导致的最坏情况”，解决不了“大量重复元素导致的分区失衡”。

  • 普通二路分区（Lomuto）会把数组分成< pivot和>= pivot两部分，当数组里全是相同元素时，所有元素都会被分到右边，每次分区只能排除1个元素，算法直接从O(n)退化成O(n²)，必然超时。
  • 这不是代码写错了，是二路分区的逻辑本身，在重复元素场景下就有致命短板。

3. 核心解法：三路划分是解决重复元素退化的终极方案

要解决重复元素的问题，核心思路是把“两段分区”改成“三段分区”，也就是荷兰国旗算法：

  • 把数组分成三段：< pivot、== pivot、> pivot，把所有和pivot相等的元素都聚在中间。
  • 这样哪怕全是重复元素，一次分区就能直接确定目标下标是否在中间段，直接返回结果，不会再一层层递归/迭代，完美解决退化问题。
  • 同时，分区后只需要去左边或右边继续找，中间的重复元素全部跳过，效率远高于二路分区。

4. 细节认知：for循环和while循环的适用边界完全不同

我想把三路划分的while改成for循环，本质是没搞懂两种循环的核心差异：

  • for循环的索引是自动、规则递增的，适合“每个元素只检查一次，不需要回头”的场景，比如二路分区里的j，每个元素只判断一次，不会回头。
  • while循环的索引是完全可控的，适合“索引增长不规则，有时需要暂停、甚至重新检查当前元素”的场景。三路划分里，当nums[i] > pivot时，交换过来的新元素还没检查，i不能前进，必须留在原地重新判断，这是for循环做不到的。

思路迁移范式：从“连续踩坑”到“全场景稳定写法”

我的错误思路演进（三次踩坑）

  1. 第一次错误：pivot取值位置错误 写了二路分区的框架，但交换pivot后依然从旧下标取值，导致partition逻辑完全错误，基础用例都跑不通。
  2. 第二次错误：二路分区无法应对重复元素 修正了pivot取值，普通用例能过，但遇到大量重复元素的用例直接超时，不知道问题根源在哪，以为是随机化没写好。
  3. 第三次错误：强行改写循环结构 知道要用三路划分解决超时，但想强行统一成for循环写法，忽略了“交换后需要重新检查当前元素”的逻辑，导致分类错误、结果不对。

最优解新思路（全场景兼容，面试首选）

严格按照“预判边界→选对方案→严谨实现”的思路，一步到位写出稳定的代码：

  1. 先做需求与边界预判：题目没说数组元素唯一，必须考虑大量重复元素的场景，直接放弃二路分区，选择三路划分。
  2. 明确三路划分的核心不变量：维护三个指针lt、i、gt，把数组分成四段，全程坚守区间规则。
  3. 用while循环做可控的索引遍历：根据元素和pivot的大小关系，决定索引是前进、后退还是原地不动，保证每个元素都被正确分类。
  4. 分区后精准缩小区间：根据目标下标和lt、gt的关系，直接决定去左边、右边，还是直接返回结果，完全跳过中间的重复元素。
