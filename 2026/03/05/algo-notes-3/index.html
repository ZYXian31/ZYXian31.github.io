📌 小知识点：快慢指针中点落点速查

1) 术语统一

设链表长度为 n，下标从 0 开始：

  • 左中点：(n-1)//2（n=4 → index 1；n=5 → index 2）
  • 右中点：n//2（n=4 → index 2；n=5 → index 2，奇数时左右相同）

2) 两种 while 条件对应左/右中点

A. while fast and fast.next → slow 停在右中点（偶数取右，奇数取中）

例子：1→2→3→4 时 slow 在 3（index 2）；1→2→3→4→5 时 slow 在 3（index 2）。 适用：后半段从 slow 开始。

B. while fast.next and fast.next.next → slow 停在左中点（偶数取左，奇数取中）

例子：1→2→3→4 时 slow 在 2（index 1）；1→2→3→4→5 时 slow 在 3（index 2）。 适用：后半段从 slow.next 开始，典型：回文链表、reorder list。

3) 需要前驱时：用 prev 版本

适用：需要 prev.next = None 断开链表（归并排序递归版常用）。

4) fast 从 head.next 开始变体

slow 更偏左侧，适用于需要 slow 停在"前半段末尾"的场景。

5) 速查表

模板 while 条件 fast 初始 slow 落点（偶数） slow 落点（奇数） 常见用途
右中点模板 fast and fast.next head 右中点 中点 后半段从 slow 开始
左中点模板 fast.next and fast.next.next head 左中点 中点 后半段从 slow.next 开始（回文/重排）
右中点 + 前驱 fast and fast.next head prev=左中点 prev=中点前驱 需要断链拿前驱
fast=head.next 变体 fast and fast.next head.next 更偏左 中点附近 控制偶数长度偏移

6) 选择规则（最实用）

先决定：我打算用 slow 还是 slow.next 作为后半段头？

  • 用 slow 作为后半段头 → 右中点模板（fast and fast.next）
  • 用 slow.next 作为后半段头 → 左中点模板（fast.next and fast.next.next）
  • 需要断链且要前驱 → prev 版本最稳

题目三：环形链表 II（Linked List Cycle II）

题目描述

返回入环节点；无环返回 None。

我的思考背景

我能背代码，但不理解为什么"相遇后把 fast 放回 head，再同步走就到入口"。

关键推导

设：

  • a：head 到入口的距离
  • b：入口到相遇点的距离
  • c：相遇点回到入口的距离（即环长 - b）
  • 环长 = b + c

第一阶段：fast 与 slow 相遇

（这里"步"指经过的边数，即 next 指针的跳转次数，不是节点数。a、b、c 均为边数。）

slow 走了 a + b 步；fast 走了 a + b + k*(b+c) 步（多绕了 k 圈）。

由 fast = 2 * slow：

因此：a ≡ c (mod 环长)

第二阶段：为什么同步走能到入口？

把 fast 重置到 head，slow 留在相遇点：

  • fast 走 a 步到达入口
  • slow 走 a 步 = 走 c + (k-1)*(b+c) 步，即从相遇点走 c 到入口，再绕 k-1 圈回到入口

两者同时抵达入口，相遇即为答案。

直觉版补充

第一阶段：环里 fast 比 slow 每步多走 1，相对速度差为 1，必追上（最多绕一圈）。

第二阶段：相遇点距入口的"剩余距离 c"恰好等于 head 到入口的距离 a（模环长），所以两人从各自起点同步走，必在入口汇合。

题目六：合并 K 个有序链表（Merge k Sorted Lists）

题目描述

合并 k 条升序链表为一条升序链表。

我的两种思路

  • 堆：O(n log k) 时间 / O(k) 空间

  • 分治两两合并：O(n log k) 时间 / O(1) 额外空间（不算 lists 数组本身）

    思路：把 k 条链表两两配对合并，每轮把链表数量减半，共 log k 轮。每轮合并的总节点数为 n，所以整体 O(n log k)。与堆相比不需要额外的优先队列，空间更省。

我写的原地两两合并（最终版、简洁漂亮）

核心思路：不新建数组，直接在 lists 原地压缩结果。

  • use_len：当前有效长度（每轮减半）
  • 每轮 for i in range(0, use_len, 2)：把 lists[i] 和 lists[i+1] 合并，写回 lists[cur_idx]，cur_idx 自增
  • 奇数条时最后一条 lists[i+1] 不存在，用 None 代替（merge 遇到 None 直接返回另一条）
  • 每轮结束后 use_len = cur_idx，即新的有效长度

为什么是 O(1) 额外空间？ 结果直接原地写回 lists[0..cur_idx-1]，没有新建任何辅助数组（merge 函数本身只用了一个 dummy 节点，O(1)）。

题目七：LRU Cache

题目描述

实现 LRU，要求 get/put 平均 O(1)。

我的思路是否合理/最优？

合理且最优（经典解）：

  • dict：key → node
  • 双向链表：维护使用顺序（MRU 在头，LRU 在尾）

我写过的错误（真实踩坑）

第一版 LRU 的错误类型很典型：

  • 没有哑 head/tail → None 边界爆炸
  • 插入/删除没维护"四条边"
  • 淘汰时删错 key / del mp[...] 忘写 self
  • self.map vs self.mp 命名错误
  • get 命中后忘 return，导致 put 永远误判未命中

这些错误的"转换路径"是：把散落在 get/put 里的链表操作抽象成 remove/add_front，让每次都走同一条正确路径。

  • 哑 head/tail 消灭 None 分支

总结：要"记住"的是惯性思维到最优解的转换路径

今天的收获可以浓缩成一句：

记住：每类错误出现时，我应该如何把它一步一步推回正确范式。

例子：

  • 相交链表：从"计数终止"推回"状态相等终止"
  • 回文链表：从"错条件/漏推进"推回"快慢模板/反转四连击"
  • 随机链表：从"需要 dict 映射"推回"结构内嵌映射 old.next"
  • 链表排序：从"像数组一样排序"推回"split/merge 的固定流程"
  • LRU：从"手写四条边"推回"remove/add_front 两个原子操作"
